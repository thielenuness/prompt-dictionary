- ✅ Agree → will fix
- 🤔 Partially agree → will adapt
- 💡 Disagree → have a technical argument
- ❓ Need more context

**[SUGGESTED REPLY]**
Write the reply directly — professional and objective.

If arguing, cite references or justify technically.
If needing context, ask a specific question.

What needs to be done in the code (or no action needed).

not the person.

If not ready, run /create-pr first.

## Steps

1. Check authentication: `!gh auth status`
2. Confirm current branch: `!git branch --show-current`
3. Open the PR:

gh pr create \
  --title "[TYPE-0000] Change title" \
  --body "$(cat pr-description.md)" \
  --base main \
  --milestone "vX.X.X" \
  --label "bug|feature|chore" \
  --assignee "@me"

4. Display the link of the created PR.

## Labels by type
FEAT → `feature` · BUG → `bug` · CHORE → `chore`

If gh CLI is not installed:
`brew install gh && gh auth login`

kit · 01 Architecture 19 perguntas · 6 frames · 1 prompt
CTX Contexto do problema Prioridade alta
CTX · 001
Qual problema estamos resolvendo?
Objetivo de negócio antes de qualquer decisão técnica
Entender o "porquê" evita desenhar a solução certa para o problema errado. Uma resposta vaga aqui é sinal de que o problema ainda não está bem definido — e isso já é uma informação valiosa.
CTX · 002
Quem são os usuários/clientes desse sistema?
Volume, perfil técnico, localização, frequência de uso
Usuário interno de backoffice tem tolerância a latência completamente diferente de um cliente final no mobile. O perfil define requisitos não-funcionais inteiros.
CTX · 003
Qual é o estágio atual? Greenfield, migração ou refactoring?
Define a liberdade de decisão arquitetural
Greenfield permite começar do ideal. Migração exige estratégia de convivência com o legado. Refactoring tem restrições de compatibilidade. São contextos completamente distintos.
CTX · 004
Existe stack ou tecnologia já definida/obrigatória?
Evita propor algo que já está fora de cogitação
Nada mais frustrante do que elaborar uma solução brilhante em Kafka só para descobrir que a empresa já tem contrato com RabbitMQ. Pergunte antes de propor.
ESC Escala e requisitos não-funcionais Técnico
ESC · 001
Qual é a expectativa de volume de requisições/dados?
Centenas, milhares ou milhões por dia?
Ordens de magnitude importam. Uma solução para 1k req/dia e uma para 10M req/dia são arquiteturas completamente diferentes. Esteja confortável dizendo "para essa escala pensaria em X, mas para Y escala mudaria para Z".
ESC · 002
Qual a disponibilidade esperada? Existe SLA definido?
99%? 99,9%? Isso muda completamente a decisão de infra
99% = ~87h de downtime/ano. 99,9% = ~8h. 99,99% = ~52min. Cada 9 a mais custa exponencialmente mais em complexidade e infra. A resposta guia decisões de redundância, failover e DR.
ESC · 003
A carga é previsível ou tem picos? Existe sazonalidade?
Impacta estratégia de auto-scaling e provisionamento
Black Friday, final de mês, horário de pico — picos previsíveis pedem estratégias diferentes de picos aleatórios. Scaling reativo vs. proativo, warm pools, scheduled scaling.
ESC · 004
Quais são as restrições de latência aceitável?
Tempo de resposta esperado para o usuário final
P95 ou P99? Latência média esconde outliers críticos. A resposta determina se cabe cache, CDN, se a comunicação pode ser assíncrona, e onde os serviços devem ser posicionados geograficamente.
SEC Segurança e conformidade Contextual
SEC · 001
Há dados sensíveis ou regulatórios envolvidos?
LGPD, PCI-DSS, HIPAA — muda autenticação, criptografia e auditoria
Dados de cartão exigem PCI-DSS e mudam onde e como você armazena. LGPD afeta retenção, anonimização e direito ao esquecimento. Não dá para ignorar na arquitetura.
SEC · 002
Quem deve ter acesso ao quê? Existe multi-tenancy?
Define a estratégia de autenticação e autorização
Multi-tenancy muda tudo: isolamento de banco, escopos de token, billing, logs de auditoria. Um RBAC simples resolve um caso; permissões granulares com tenants exigem outro design completamente.
OPS Operação e equipe Contextual
OPS · 001
Qual é o tamanho e estrutura da equipe de engenharia?
Microsserviços com 3 devs pode ser over-engineering
Conway's Law é real: a arquitetura vai espelhar a estrutura de comunicação da equipe. A solução precisa ser sustentável para quem vai mantê-la no dia a dia.
OPS · 002
Como é o modelo de deploy hoje? Existe CI/CD?
Frequência de releases, rollback, feature flags
Uma equipe que deploya uma vez por sprint tem necessidades diferentes de uma que faz deploy contínuo. A maturidade de CI/CD atual é o ponto de partida real, não o ideal.
OPS · 003
Existe observabilidade hoje? Logs, métricas, tracing?
Se não existe, precisa entrar como requisito da solução
Sistemas sem observabilidade são caixas-pretas. Propor microsserviços sem mencionar distributed tracing é propor algo impossível de operar. Logging, métricas e tracing são requisitos de primeira classe.
INT Integrações e dependências Técnico
INT · 001
Quais sistemas externos esse serviço precisa integrar?
APIs de terceiros, legados, parceiros
Integrações com sistemas que você não controla são riscos. SLA do parceiro, disponibilidade, contratos de API, circuit breakers — tudo isso precisa aparecer na arquitetura. Ignorar integrações é a causa #1 de surpresas em produção.
INT · 002
A comunicação entre serviços é síncrona ou pode ser assíncrona?
Define se precisa de fila/mensageria ou chamadas diretas
Comunicação síncrona acopla disponibilidade — se B cai, A falha. Assíncrona desacopla, mas adiciona eventual consistency, dead-letter queues e idempotência. O negócio tolera atraso ou não?
INT · 003
Qual o modelo de dados hoje? Existe banco legado?
Impacta decisão de banco e estratégia de migração
Migrar um Oracle com 15 anos de stored procedures é um projeto à parte. A decisão de banco nova precisa conviver com o legado durante a transição — dual-write, strangler fig, event sourcing.
BIZ Restrições de negócio Prioridade alta
BIZ · 001
Existe prazo ou MVP definido?
Trade-off entre solução ideal e velocidade de entrega
Um MVP em 6 semanas pede decisões diferentes de uma plataforma com 12 meses de runway. A arquitetura ideal para longo prazo pode ser inviável para o prazo dado.
BIZ · 002
Há restrição de orçamento de infra?
Cloud gerenciado tem custo diferente de self-hosted
RDS gerenciado vs. PostgreSQL self-hosted em EC2. EKS vs. instâncias simples. Managed Kafka vs. SQS. Cada escolha tem implicação de custo e complexidade operacional.
BIZ · 003
Quais são os maiores riscos ou pontos de falha conhecidos hoje?
Geralmente revela o real problema que querem resolver
Essa pergunta quase sempre revela o problema real. "O sistema cai todo final de mês" ou "a integração com o banco demora 30s" define prioridades melhor do que qualquer requisito escrito.
